## 📋 一、项目概述

### 🏷️ 项目定位

**ChaosDroid** 是一个 **Android 故障注入测试与恢复验证平台**。

```
┌─────────────────────────────────────────────────────────────────┐
│  💡 核心价值主张                                                 │
├─────────────────────────────────────────────────────────────────┤
│  ✅ 将手动、零散的故障注入流程 → 配置化、可重复、可回归的测试平台  │
│  ✅ 验证系统在升级、重启、压力、稳定性场景下的韧性能力            │
│  ✅ 输出结构化的测试结论与复盘报告                                │
└─────────────────────────────────────────────────────────────────┘
```

### 🎓 面试高频问题 #1: "这个项目解决了什么问题?"

> **回答模板**:
> "这是一个混沌工程工具，专门针对 Android 设备。传统测试难以覆盖系统故障场景，
> ChaosDroid 通过可控的故障注入（如存储压力、低电量、网络抖动），验证系统恢复能力，
> 帮助团队提前发现生产环境可能出现的问题。"

## 🔄 五、执行流程详解

### 📈 状态机流转图

```
   ┌─────────┐
   │ QUEUED  │  📝 创建执行记录
   └────┬────┘
        ↓
   ┌─────────┐
   │PREPARING│  🔍 检查设备状态、收集设备属性
   └────┬────┘
        ↓
   ┌─────────┐
   │INJECTING│  💉 执行故障注入
   └────┬────┘
        ↓
   ┌─────────┐
   │VALIDATING│ ✅ 验证故障效果
   └────┬────┘
        ↓
   ┌─────────┐
   │RECOVERING│ 🔧 执行恢复操作
   └────┬────┘
        ↓
   ┌─────────────────────────────────────┐
   │        最终判定 (Final Judgment)      │
   ├─────────────────────────────────────┤
   │  🟢 passed  │ 🟡 partial │ 🔴 failed │
   └─────────────────────────────────────┘
```

### 🎓 面试高频问题 #4: "执行流程是如何设计的?"

> **回答模板**:
> "采用状态机模式，5 个有序状态：
> 1. **Preparing**: 设备状态检查（在线、电量、存储、启动完成）
> 2. **Injecting**: 从注册表获取注入器，执行故障注入
> 3. **Validating**: 验证故障是否生效（启动、电量、存储检查）
> 4. **Recovering**: 清理注入效果 + 执行恢复步骤
> 5. **Final Judgment**: 根据各阶段结果判定最终状态"

### 📊 最终状态判定逻辑

```
┌─────────────────────────────────────────────────────────────────┐
│  📋 判定规则                                                     │
├─────────────────────────────────────────────────────────────────┤
│  🔴 failed     → 注入失败                                        │
│  🔴 failed     → 注入成功 + 验证失败 + 恢复成功                   │
│  🟡 partial    → 注入成功 + 验证成功 + 恢复失败                   │
│  🟢 passed     → 注入成功 + 验证成功 + 恢复成功                   │
└─────────────────────────────────────────────────────────────────┘
```

## 🧩 六、设计模式应用

### 🎨 设计模式一览表

| 🏷️ 模式 | 📍 实现位置 | 💡 应用目的 |
|---------|-------------|-------------|
| **状态机模式** | `state_machine.py` | 驱动执行阶段流转 |
| **策略模式** | `BaseInjector`, `BaseValidator` | 可插拔的故障注入/验证策略 |
| **注册表模式** | `INJECTOR_REGISTRY` | 动态注入器注册与查找 |
| **工厂模式** | `_setup_executor()`, `_setup_injector()` | 根据配置创建执行器/注入器 |
| **观察者模式** | `ObservationCollector` | 收集各阶段的产物 |
| **依赖注入** | `get_settings()`, singleton | 配置与服务访问 |
| **上下文对象** | `InjectContext`, `ValidationContext` | 传递执行状态 |

### 🎓 面试高频问题 #5: "项目中使用了哪些设计模式?"

> **回答模板**:
> "项目中使用了多种设计模式：
> 1. **状态机模式**: 驱动测试执行的 5 个阶段流转，每个状态有独立的处理器
> 2. **策略模式**: `BaseInjector` 抽象基类，允许不同故障类型有独立的注入策略
> 3. **注册表模式**: `INJECTOR_REGISTRY` 实现 Plugin 式扩展，新增注入器只需注册
> 4. **工厂模式**: 根据配置动态创建 Mock 或 Real 设备执行器
> 5. **观察者模式**: 在各阶段自动收集 logcat、截图等产物"

## 💉 七、故障注入器详解

### 🔧 内置注入器列表

| 🏷️ 注入器 | 📋 功能 | ⚠️ 风险等级 |
|-----------|---------|-------------|
| `StoragePressureInjector` | 🗂️ 填充存储空间，模拟低存储场景 | 中 |
| `LowBatteryInjector` | 🔋 模拟低电量场景 | 低 |
| `NetworkJitterInjector` | 📡 网络延迟、断网模拟 | 低 |
| `RebootTimeoutInjector` | 🔄 模拟启动超时 | 高 |
| `CpuIoStressInjector` | ⚡ CPU 和 IO 压力测试 | 中 |
| `MonkeyStabilityInjector` | 🐵 Monkey 测试，收集 Crash/ANR | 高 |

### 🔨 注入器生命周期

```python
class BaseInjector:
    async def prepare(context)   # 📋 前置检查
    async def inject(context)    # 💉 执行注入
    async def cleanup(context)   # 🧹 清理效果
```

### 🎓 革新式问题 #6: "如何扩展新的故障类型?"

> **回答模板**:
> "继承 `BaseInjector`，实现三个生命周期方法，然后注册到 `INJECTOR_REGISTRY`：
> ```python
> @register_injector('custom_fault')
> class CustomInjector(BaseInjector):
>     async def prepare(self, ctx): ...
>     async def inject(self, ctx): ...
>     async def cleanup(self, ctx): ...
> ```
> 注册表模式使得扩展无需修改核心代码，符合开闭原则。"

## 🔐 九、并发与安全设计

### 🔒 设备锁管理

```
┌─────────────────────────────────────────────────────────────────┐
│  🛡️ DeviceLockManager                                            │
├─────────────────────────────────────────────────────────────────┤
│  🎯 目的: 防止同一设备被多个测试同时执行                          │
│  🔧 实现: asyncio.Lock + 设备 ID 映射                            │
│  📋 API:                                                         │
│     - acquire(device_id)  → 获取锁                               │
│     - release(device_id)  → 释放锁                               │
│     - is_locked(device_id) → 检查状态                            │
└─────────────────────────────────────────────────────────────────┘
```

### 🎓 革新式问题 #8: "如何处理并发执行的安全问题?"

> **回答模板**:
> "同一设备上并发执行会导致状态混乱。项目实现了 `DeviceLockManager`，
> 使用 `asyncio.Lock` 为每个设备 ID 维护独立锁。
> 执行前必须 `acquire()`，完成后 `release()`，确保互斥访问。
> 这避免了恢复操作与注入操作冲突的问题。"

## 📊 十、数据模型设计

### 🗃️ 核心实体关系

```
ScenarioTemplate (场景模板)
    │
    ├── contains ──→ FaultProfile (故障配置)
    │                   ├── fault_type: str
    │                   ├── params: dict
    │                   └── risk_level: str
    │
    ├── contains ──→ ValidationProfile (验证配置)
    │                   ├── check_boot: bool
    │                   ├── check_battery: bool
    │                   └── check_storage: bool
    │
    ├── contains ──→ RecoveryProfile (恢复配置)
    │                   ├── auto_recover: bool
    │                   └── recovery_steps: list
    │
    └── creates ──→ ScenarioRun (执行记录)
                      ├── status: RunStatus
                      ├── started_at: datetime
                      ├── finished_at: datetime
                      │
                      └── has ──→ ScenarioStep (步骤记录)
                      │               ├── phase: str
                      │               ├── result: str
                      │               └── details: dict
                      │
                      └── produces ──→ Artifact (产物)
                      │                   ├── type: str
                      │                   ├── path: str
                      │
                      └── generates ──→ Report (报告)
                                          ├── content: str
                                          ├── format: str
```

### 🎓 面试高频问题 #9: "数据模型是如何设计的?"

> **回答模板**:
> "采用 SQLAlchemy 2.0 异步 ORM。核心实体：
> 1. **ScenarioTemplate**: 可复用的测试场景配置
> 2. **ScenarioRun**: 单次执行记录，跟踪状态变化
> 3. **ScenarioStep**: 每个阶段（Preparing/Injecting/...）的详情
> 4. **Artifact**: 收集的产物（logcat、截图、dumpsys）
> 5. **Report**: 生成的测试报告

> 模板与执行记录分离，支持场景配置复用；每次执行独立记录，便于追溯。"

## 🚀 十四、快速上手命令

### ⌨️ CLI 常用命令

```bash
# 🏁 初始化项目
chaosdroid init

# 🌐 启动 Web 服务
chaosdroid serve --port 8000

# 📋 场景管理
chaosdroid scenario list              # 列出所有场景
chaosdroid scenario create --name X   # 创建场景
chaosdroid scenario show ID           # 查看详情

# ▶️ 执行测试
chaosdroid run create --scenario-id ID  # 创建执行
chaosdroid run start ID                  # 开始执行
chaosdroid run status ID                 # 查看状态

# 📱 设备管理
chaosdroid device list               # 列出设备
chaosdroid device status DEVICE_ID   # 设备状态

# 📊 报告查看
chaosdroid report list              # 列出报告
chaosdroid report show RUN_ID       # 查看报告
chaosdroid report export RUN_ID     # 导出报告
```
