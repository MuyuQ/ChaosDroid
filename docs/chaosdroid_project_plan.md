# ChaosDroid 项目规划书

## 1. 项目概述

`ChaosDroid` 是一个面向安卓设备的故障注入测试与恢复验证平台。它的目标不是被动分析问题，而是主动在可控条件下制造异常，验证系统在升级、重启、压力和稳定性场景中的鲁棒性、恢复能力和可观测性。

项目的核心价值在于把原本依赖人工经验和零散脚本的“异常环境构造”流程，抽象为可配置、可重复、可回归的测试资产。相比普通自动化测试项目，它更贴近测试开发岗位的真实工作内容，也更能体现你对安卓升级链路、稳定性测试和机房执行场景的理解。

## 2. 项目定位

### 2.1 对外定位

项目定位为：

- 安卓故障注入测试平台
- 升级与稳定性鲁棒性验证工具
- 面向测试开发的轻量级移动端混沌工程平台

### 2.2 目标用户

- 测试开发工程师
- 系统稳定性测试工程师
- 升级验证与回归测试负责人
- 需要批量验证设备恢复能力的质量团队

### 2.3 简历叙事

简历中的项目价值建议表述为：

- 将人工构造故障环境的流程平台化
- 将升级前、升级中、升级后常见异常场景标准化
- 验证系统在异常条件下的恢复能力和稳定性
- 输出结构化结论与复盘报告

## 3. 项目要解决的问题

在安卓升级与稳定性测试中，很多高价值问题不是常规功能缺陷，而是异常条件下才会暴露的鲁棒性问题，例如：

- 升级包下载过程中网络中断
- 升级前或升级中设备存储空间不足
- 升级后重启等待超时或 boot 卡死
- monkey 压测叠加资源压力后触发 crash 或 ANR
- 低电量、高负载、I/O 压力下升级表现退化
- 设备在故障发生后能否自动恢复，是否需要人工介入

传统做法的问题在于：

- 场景构造分散在多个脚本里，难复用
- 故障注入时机依赖人工控制，难稳定复现
- 执行结果缺乏统一结构，难横向比较
- 故障恢复没有明确标准，复盘信息不完整

`ChaosDroid` 通过场景定义、注入引擎、执行编排、结果判定和报告输出，将这条测试链路系统化。

## 4. 核心目标

第一版项目目标固定如下：

- 支持定义标准化故障注入场景
- 支持在指定阶段执行故障注入
- 支持真实设备和模拟设备两种模式
- 支持执行验证步骤并收集观测信息
- 支持执行恢复动作并判断恢复结果
- 支持生成可读的 Markdown/HTML 报告

## 5. 非目标

第一版不做以下内容：

- 不实现 Android 底层源码改动
- 不实现真正的内核级 COW 改造
- 不实现复杂的分布式调度平台
- 不实现大而全的权限系统
- 不覆盖所有厂商定制 ROM

项目重点是“故障注入测试产品化”，而不是“安卓底层开发”。

## 6. 核心能力设计

## 6.1 场景库

平台以“场景”为最小管理单位，每个场景包括：

- 测试目标
- 注入阶段
- 故障类型
- 注入参数
- 验证步骤
- 恢复步骤
- 判定标准

场景应可保存、复用、克隆和批量执行。

## 6.2 阶段化注入

故障注入不是任意时刻触发，而是绑定到固定阶段：

- `precheck`
- `prepare`
- `upgrading`
- `reboot_wait`
- `post_boot`
- `post_validate`

这种设计可以保证：

- 故障触发点可复现
- 报告中的结论有上下文
- 同一类场景可以横向比较

## 6.3 恢复能力验证

测试结果不只看是否失败，还要看：

- 故障是否被正确感知
- 系统是否自动恢复
- 恢复耗时是否可接受
- 恢复后能否继续执行验证步骤
- 是否需要人工清理或人工干预

## 6.4 结果结构化

每次执行结果必须标准化输出：

- 场景名称
- 设备信息
- 注入点
- 注入动作
- 验证结果
- 恢复结果
- 风险等级
- 关键证据
- 建议动作

## 7. 推荐故障注入类型

第一版建议只做 6 类场景，保证项目足够扎实。

### 7.1 存储压力注入

通过向指定目录写入大文件模拟存储空间不足。

适用场景：

- 升级包下载
- 升级包解压
- 升级前环境检查

验证重点：

- 升级是否被安全拒绝
- 错误是否可感知
- 清理后是否恢复

### 7.2 低电量场景

通过模拟低电量条件阻断任务。

验证重点：

- 前置检查是否生效
- 任务是否安全退出
- 状态是否被正确标记

### 7.3 网络波动场景

在包下载或在线验证阶段制造断网、限速或抖动。

验证重点：

- 是否重试
- 失败是否可解释
- 网络恢复后是否能继续

### 7.4 重启等待异常

在升级后重启阶段模拟超时、掉线、启动未完成。

验证重点：

- 设备是否能重新上线
- `boot_completed` 是否置位
- 是否进入不可恢复状态

### 7.5 CPU / I/O 压力场景

通过压测任务制造系统负载，观察升级或稳定性行为是否退化。

验证重点：

- 启动耗时是否明显增加
- monkey 结果是否恶化
- 服务是否超时

### 7.6 monkey 稳定性场景

在升级后执行 monkey，并叠加其他资源压力。

验证重点：

- crash / ANR 数量
- 任务完成率
- 系统是否仍可用

## 8. 系统形态

项目最终形态建议为 `Web 平台 + CLI/worker`：

- Web：用于场景配置、结果查看、报告展示
- CLI：用于本地调试、批量执行、与设备交互
- Worker：用于后台执行故障注入与验证编排

这种形态既保留平台感，也符合你的 Python 技术栈和使用习惯。

## 9. 技术栈建议

- `FastAPI`
  - API 与页面服务
- `Typer`
  - 命令行入口
- `SQLAlchemy`
  - 数据模型
- `SQLite`
  - 单机存储
- `Jinja2 + HTMX`
  - 轻量页面
- `Pydantic`
  - 场景配置和结果校验
- `subprocess`
  - 调用 `adb`、`cmd`、`monkey`、shell 命令

## 10. 项目模块

### 10.1 场景管理模块

负责场景定义、版本管理、克隆与启停。

### 10.2 故障注入引擎

负责真正执行注入动作。

### 10.3 执行编排模块

负责把“前置检查 -> 注入 -> 验证 -> 恢复 -> 采集”串起来。

### 10.4 观测采集模块

负责抓取日志、属性、结果文件和阶段事件。

### 10.5 结果判定模块

负责判断注入是否生效、系统是否恢复、任务是否通过。

### 10.6 报告模块

负责输出 HTML/Markdown 报告。

## 11. 第一版范围控制

为了保证项目能真正做完，第一版范围固定为：

- 只做升级后稳定性故障注入验证主线
- 只做 6 类 fault profile
- 只支持单机部署
- 只支持单设备执行和批量顺序执行
- 只做轻 Web，不做重前端

如果时间允许，再扩展：

- 批量设备并发执行
- 更复杂的资源调度
- 更多恢复策略
- 更细的性能基线对比

## 12. 面试表达重点

这个项目最适合面试时讲的角度有三个：

### 12.1 主动测试而不是被动验证

你不是只在跑回归，而是在主动制造异常环境验证系统鲁棒性。

### 12.2 平台化沉淀而不是脚本堆积

你不是写了几个临时脚本，而是把场景、注入、验证、恢复和报告抽象成了稳定平台。

### 12.3 系统理解和工程落地并重

你理解安卓升级、重启、压力、稳定性链路，同时也把它们用 Python 做成了一个可执行系统。

## 13. 简历描述建议

可以写成：

- 设计并实现安卓设备故障注入与恢复验证平台，支持在升级前、升级中和升级后阶段注入存储压力、网络波动、重启异常、低电量与 monkey 稳定性场景
- 基于 Python/ADB 构建 fault profile、执行编排、观测采集与结果判定能力，自动输出异常观测、恢复结论与风险报告
- 将人工构造异常环境与稳定性复测流程平台化，提升升级验证、稳定性测试与故障复现效率

## 15. 结论

`ChaosDroid` 非常适合作为你的简历项目，因为它同时满足三个条件：

- 和你真实经历高度匹配
- 项目方向不烂大街

它不是普通后台，也不是泛化自动化测试工具，而是一套更贴近真实测试价值的异常验证平台。
